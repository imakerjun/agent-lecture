# FE-BE 원정대 미션 설계서 (run-01)

## 1. 미션 개요

**한 줄 요약**: 프론트엔드 크루 1명과 백엔드 크루 1명이 페어로, 우테코 크루들이 실제 사용할 수 있는 웹 애플리케이션을 설계·개발·배포하고, 레벨2 기간 동안 유지보수한다.

### 핵심 경험

- **FE-BE 경계를 넘나드는 협업**: API 명세를 함께 설계하고, 서로의 코드를 리뷰하며, 웹 애플리케이션의 전체 흐름을 체감한다.
- **TDD 기반 개발**: 프론트엔드 크루는 백엔드 테스트 작성을, 백엔드 크루는 프론트엔드 테스트 작성을 경험하며, 서로의 영역에서 TDD가 어떻게 작동하는지 직접 확인한다.
- **배포 후 유지보수**: 실사용자의 피드백을 바탕으로 우선순위를 정하고, 개선 사이클을 돌린다.
- **레벨1 송곳의 확장**: 레벨1 원정대에서 갈고닦은 기술적 강점(송곳)을 이번 미션에 의도적으로 적용하고, 페어에게 전파한다.

## 2. 단계별 구성

### 1단계: 기획 및 설계 (원정대 1~3일차)

**목적**: 무엇을 만들지 합의하고, FE-BE 간 계약(API 명세)을 확정한다.

**활동**:
- 페어가 함께 서비스 아이디어를 선정한다. 대상 사용자는 "우테코 크루"로 한정한다.
- 핵심 기능 3개 이내로 범위를 좁힌다.
- API 명세(엔드포인트, 요청/응답 형식)를 문서로 작성한다.
- 데이터 모델을 함께 설계한다. 프론트엔드 크루도 ERD 작성에 참여한다.
- 레벨1 원정대에서 각자 만든 "송곳"을 공유하고, 이번 미션에 적용할 방법을 논의한다.

**산출물**:
- 서비스 기획서 (1페이지: 대상 사용자, 핵심 기능, 해결하려는 문제)
- API 명세 문서
- 데이터 모델 (ERD 또는 스키마)
- 송곳 적용 계획 (각자 1개 이상)

### 2단계: 핵심 기능 개발 (원정대 4~8일차)

**목적**: MVP를 구현하여 배포 가능한 상태로 만든다.

**활동**:
- 백엔드: API 서버 구현. 테스트를 먼저 작성한 뒤 프로덕션 코드를 작성한다.
- 프론트엔드: UI 구현. 1단계에서 합의한 API 명세를 기반으로 MSW 등을 활용해 독립적으로 개발을 시작한다.
- **교차 테스트 작성**: 프론트엔드 크루가 백엔드 테스트 1개 이상, 백엔드 크루가 프론트엔드 테스트 1개 이상을 직접 작성한다. 이때 페어가 옆에서 가이드한다.
- 매일 15분 이내의 페어 체크인을 진행한다: 진행 상황 공유 + API 변경 사항 동기화.

**산출물**:
- 핵심 기능이 동작하는 MVP
- 교차 테스트 코드 (FE→BE 테스트, BE→FE 테스트 각 1개 이상)
- 일일 체크인 기록 (간단한 메모)

### 3단계: 배포 및 사용자 테스트 (원정대 9~10일차)

**목적**: 실제 환경에 배포하고, 우테코 크루들에게 피드백을 수집한다.

**활동**:
- FE/BE를 각각 배포한다. 배포 환경은 크루가 자유롭게 선택한다.
- 우테코 크루 5명 이상에게 서비스를 사용해달라고 요청하고, 구두 또는 서면으로 피드백을 수집한다.
- 수집한 피드백을 기능 개선, 버그 수정, UX 개선으로 분류한다.
- 원정대 발표에서 서비스 데모와 함께 기술적 의사결정 과정을 공유한다.

**산출물**:
- 배포된 서비스 URL (FE + BE)
- 사용자 피드백 정리 문서
- 원정대 발표 자료

### 4단계: 유지보수 (레벨2 남은 기간, 약 4~6주)

**목적**: 실사용자 피드백을 기반으로 서비스를 개선하며, 유지보수 경험을 쌓는다.

**활동**:
- 3단계에서 수집한 피드백과 자체 회고를 바탕으로 이슈 목록을 생성한다.
- 2주 단위 스프린트로 개선 작업을 진행한다. 매 스프린트 시작 시 이슈 우선순위를 함께 결정한다.
- 스프린트마다 최소 1회 배포한다.
- 마지막 스프린트 종료 시 회고 문서를 작성한다.

**산출물**:
- 이슈 목록 및 우선순위 (GitHub Issues 또는 유사 도구 활용)
- 스프린트별 변경 로그
- 최종 회고 문서

## 3. 요구사항

### 기능 요구사항

#### 필수

- 우테코 크루가 실제로 사용할 수 있는 웹 서비스를 개발한다.
- 최소 1개의 CRUD(Create, Read, Update, Delete) 기능을 포함한다.
- 프론트엔드와 백엔드가 HTTP API로 통신한다.
- 사용자가 데이터를 입력하고 저장된 결과를 확인할 수 있다.

#### 선택

- 사용자 인증/인가 기능
- 실시간 업데이트 (WebSocket 등)
- 외부 API 연동

### 비기능 요구사항

#### 필수

- **테스트**: 백엔드 API에 대한 테스트와 프론트엔드 주요 로직에 대한 테스트를 작성한다.
- **교차 테스트**: 프론트엔드 크루가 백엔드 테스트 1개 이상, 백엔드 크루가 프론트엔드 테스트 1개 이상을 작성한다.
- **API 문서화**: API 명세를 README 또는 별도 문서에 기록한다.
- **배포**: 원정대 기간 내에 외부에서 접근 가능한 URL로 배포한다.
- **코드 리뷰**: 모든 PR에 대해 페어가 리뷰한다. FE 코드는 BE 크루가, BE 코드는 FE 크루가 리뷰한다.

#### 선택

- CI/CD 파이프라인 구축
- 에러 모니터링
- 성능 최적화

## 4. 협업 규칙

### 커뮤니케이션 구조

| 활동 | 빈도 | 형식 | 내용 |
|------|------|------|------|
| 페어 체크인 | 매일 1회 (15분 이내) | 대면 또는 화상 | 진행 상황, API 변경 사항, 막힌 점 공유 |
| 코드 리뷰 | PR마다 | GitHub PR 코멘트 | 상대 영역 코드 리뷰 (질문 위주도 가능) |
| 스프린트 계획 | 2주마다 | 대면 | 이슈 우선순위 결정, 작업 분배 |
| 중간 회고 | 원정대 종료 시 | 문서 작성 | 잘된 점, 개선할 점, 배운 점 |

### FE-BE 협업 포인트

1. **API 명세 공동 설계**: 프론트엔드가 필요한 데이터 형태와 백엔드가 제공 가능한 구조를 맞춰본다. 한쪽이 일방적으로 정하지 않는다.
2. **교차 코드 리뷰**: 상대 영역의 PR을 리뷰한다. 코드를 완벽히 이해하지 못해도, "이 부분이 무슨 역할인지 모르겠다"는 질문 자체가 가치 있다.
3. **교차 테스트 작성**: 서로의 영역에서 테스트를 작성한다. 이때 해당 영역의 크루가 페어 프로그래밍 방식으로 가이드한다.
4. **장애 대응 공동 참여**: 배포 후 버그가 발생하면 FE/BE 중 어디 문제인지 함께 추적한다. 로그, 네트워크 탭, 에러 메시지를 함께 확인한다.

### 갈등 해결

- 기술적 의사결정에서 합의가 안 되면, 각자의 근거를 문서로 작성한 뒤 코치에게 의견을 구한다.
- 작업 속도 차이가 발생하면, 빠른 쪽이 느린 쪽의 작업을 페어 프로그래밍으로 돕는다.

| 대상 사용자 | 우테코 크루 | 실사용자 피드백 수집 가능성을 확보하기 위함 |
| 배포 | 원정대 기간 내 최소 1회 배포 | 배포 경험을 보장하기 위함 |

### 크루 결정 영역

| 항목 | 선택지 예시 | 결정 시 고려사항 |

| FE 프레임워크 | React, Vanilla JS, Vue 등 | 레벨1에서 배운 것과의 연결 |

| 테스트 도구 | Jest, Vitest, JUnit, Cypress 등 | 기존 학습 경험과의 연결 |

## 6. 유지보수 가이드

### 유지보수 사이클

```
피드백 수집 → 이슈 등록 → 우선순위 결정 → 구현 → 코드 리뷰 → 배포 → 피드백 수집
```

### 스프린트 운영

- 2주 단위 스프린트로 운영한다.
- 스프린트 시작 시 페어가 함께 이슈 우선순위를 정한다.
- 우선순위 기준: 사용자 영향도 > 기술 부채 > 새 기능.
- 스프린트마다 최소 1회 배포한다.

### 피드백 채널

- 서비스 내 피드백 폼 (선택)
- 슬랙 채널에 사용 후기 요청 (필수)
- 주 1회 사용자 5명에게 직접 의견을 묻는다 (필수)

### 유지보수 기간의 학습 포인트

- **이슈 분류**: 사용자가 보고한 문제가 FE 이슈인지 BE 이슈인지 함께 판단한다.
- **회귀 방지**: 버그 수정 시 해당 케이스에 대한 테스트를 추가한다.
- **기술 부채 관리**: 원정대 기간에 급히 만든 코드를 리팩터링한다.

### 유지보수 종료 조건

- 레벨2 종료 시점에 최종 회고 문서를 작성한다.
- 회고에는 다음 항목을 포함한다: 전체 변경 이력 요약, 사용자 반응 추이, 기술적으로 배운 점, 협업에서 배운 점.

## 7. 평가 기준

크루에게 아래 기준을 사전에 공개한다.

### 과정 평가 (비중 60%)

| 항목 | 관찰 포인트 |
|------|------------|
| 협업 | API 명세 공동 설계 여부, 교차 코드 리뷰 진행 여부, 매일 체크인 기록 |
| TDD 실천 | 최소 1개 기능에서 테스트 선작성 흔적 (커밋 히스토리로 확인) |
| 교차 테스트 | 상대 영역 테스트 작성 여부 및 페어 프로그래밍 기록 |
| 송곳 적용 | 레벨1 송곳을 어떻게 활용했는지 기술한 문서 |

### 결과 평가 (비중 40%)

| 항목 | 관찰 포인트 |
|------|------------|
| 배포 상태 | 외부에서 접근 가능한 URL, 핵심 기능 동작 여부 |
| 테스트 커버리지 | 주요 기능에 대한 테스트 존재 여부 (커버리지 수치 기준 없음) |
| 유지보수 활동 | 스프린트별 변경 로그, 이슈 처리 이력 |
| 사용자 피드백 반영 | 피드백 → 개선으로 이어진 사례 존재 여부 |

### 평가하지 않는 것

- 서비스의 완성도나 기능 수
- 특정 기술 스택 사용 여부
- 코드 라인 수
- 디자인의 심미성

| 학습 목표 달성 가능성 | 4 | 기대효과 5개 중 대부분이 미션 활동과 대응된다. 웹 전체 이해→FE/BE 공동 설계+교차 테스트. BE 구조 이해→ERD 참여+교차 코드 리뷰. TDD→테스트 선작성 필수. 송곳 연결→송곳 적용 계획 단계 포함. | "각 기대효과에 대응하는 명시적 체크포인트"가 부족하다. 기대효과 5개를 체크리스트로 만들고, 각 단계 종료 시 "이 기대효과를 경험했는가?"를 자가 점검하는 장치를 넣으면 5점에 가까워진다. 특히 "FE만으로 경험하기 어려운 TDD/객체지향 고민"이 교차 테스트 1개로 충분히 달성되는지 의문이다. |
| 단계 설계의 적절성 | 4 | 4단계(기획→개발→배포→유지보수)가 자연스럽게 이어지고, 각 단계의 목적/산출물/기간이 명확하다. 1단계의 API 명세가 2단계 개발의 기반이 되고, 3단계 피드백이 4단계 유지보수의 입력이 되는 구조가 좋다. | 2단계(5일)에서 MVP 완성이 가능한지 현실성 검토가 필요하다. FE-BE 모두 익숙하지 않은 영역을 다루면서 5일 안에 배포 가능한 MVP를 만들기는 빠듯하다. 2단계를 "핵심 1개 기능 완성"으로 범위를 더 좁히거나, 기간을 조정하는 것을 고려할 수 있다. |
| 요구사항의 구체성과 자율성 균형 | 4 | "지정 사항"과 "크루 결정 영역"을 표로 명확히 분리한 점이 좋다. 각 지정 사항에 "이유"를 명시하여 크루가 "왜 이런 제약이 있는지" 이해할 수 있다. 서비스 주제, 기술 스택, 배포 환경 등 핵심 선택은 크루에게 위임했다. | 자율 영역에서의 의사결정이 "학습으로 이어지는 구조"가 아직 약하다. 기술 스택 선정 시 "왜 이 기술을 선택했는지" 근거를 문서로 남기도록 요구하면, 선택 행위 자체가 학습이 된다. "결정 시 고려사항"이 있지만 산출물로 요구되지는 않는다. |
| 유지보수 경험의 실질성 | 4 | 실사용자(우테코 크루) 피드백을 필수로 요구하고, 2주 스프린트, 우선순위 결정, 배포 사이클이 설계되어 있다. "주 1회 사용자 5명에게 직접 의견을 묻는다"는 구체적이고 실행 가능하다. | "사용자 피드백 → 우선순위 결정 → 구현 → 배포"의 전체 사이클이 설계서에 존재하므로 4점이다. 5점을 위해서는 이 사이클이 1회가 아닌 반복적으로 돌아간다는 것을 더 구조적으로 보장해야 한다. 예: 스프린트 종료 시 "사용자 5명에게 변경된 기능 재확인" 같은 피드백 재수집 단계를 명시한다. |

### 종합

- **총점: 20/25**
- **판정: 실험에 바로 사용 가능**
- **우선 개선 영역**:
  1. **학습 목표 체크포인트**: 기대효과 5개를 각 단계 종료 시 자가 점검하는 체크리스트로 만들어, 경험이 우연이 아닌 구조에 의해 보장되도록 한다.
  2. **교차 경험의 깊이**: 교차 테스트 "1개 이상"은 형식적 이행 가능성이 높다. 최소 수량보다는 "페어 프로그래밍으로 함께 작성"하는 과정 자체를 필수로 명시하고, 그 과정에서 배운 점을 기록하도록 요구한다.
  3. **2단계 일정의 현실성**: 5일 안에 MVP 완성이 빠듯할 수 있다. 범위를 더 좁히거나 "핵심 1개 기능만 동작"을 명시적 목표로 설정한다.

## Crew Perspective 평가

### 첫 인상

백엔드 크루와 함께 실제 서비스를 만든다는 점이 흥미롭다. 그런데 교차 테스트, 매일 체크인, 코드 리뷰 등 해야 할 것이 많아 보인다. "우테코 크루가 실제로 사용할 수 있는 서비스"라는 조건이 부담도 되지만 동기 부여도 된다.

### 체크리스트 응답

| 항목 | 반응 | 구체적 지점 |
|------|------|------------|
| 이해도 | 명확 | 단계별 구성이 4단계로 나뉘어 있고, 각 단계에서 뭘 해야 하는지 산출물이 명시되어 있어서 파악하기 쉽다. 다만 "송곳 적용 계획"이 정확히 뭘 의미하는지 처음엔 모호했다. 레벨1 원정대를 안 거친 크루가 읽으면 이해 못 할 수 있다. |
| 동기 | 재밌겠다 쪽 | 실제 크루들이 쓸 서비스를 만든다는 점, 배포 경험을 할 수 있다는 점이 동기 부여가 된다. 점심 메뉴 추천, 스터디 매칭 같은 예시도 구체적이라 아이디어가 떠오른다. |
| 불안 | 중간 | 백엔드 테스트를 내가 작성해야 한다는 게 가장 불안하다. JUnit이나 Spring 테스트를 한 번도 써본 적이 없는데 "1개 이상"이라고 해도 어디서 시작해야 할지 감이 안 온다. "페어가 가이드한다"고 했지만, 실제로 어떤 수준의 가이드를 받을 수 있는지 불확실하다. |
| 협업 걱정 | 보통 | 매일 15분 체크인, PR 코드 리뷰 등 구조가 있어서 "어떻게 소통하지?"라는 막막함은 적다. 다만 API 명세를 "공동 설계"한다고 했는데, 내가 백엔드 구조를 잘 모르는 상태에서 얼마나 기여할 수 있을지 걱정된다. 백엔드 크루가 주도하고 나는 따라가기만 하게 되지 않을까. |
| 성장 기대 | 느껴진다 | "이 미션을 끝내면 백엔드와 대화할 수 있겠다"는 기대가 생긴다. 특히 교차 코드 리뷰와 교차 테스트를 통해 상대 영역의 코드를 읽어보는 경험은 레벨3 프로젝트에서 확실히 도움이 될 것 같다. |

### 가장 걱정되는 것 (Top 3)

1. **백엔드 테스트 작성**: JUnit이나 BE 테스트 프레임워크를 전혀 모르는 상태에서 테스트를 작성해야 한다. 가이드 자료나 최소 환경 셋업이 없으면 많은 시간을 환경 구성에 허비할 수 있다.
2. **5일 안에 MVP 완성**: 서비스 기획, API 설계에 3일을 쓰고 나면 실제 코딩은 5일인데, 처음 해보는 협업 + 처음 쓰는 도구 조합으로 MVP를 만들 수 있을까. 범위 조절을 잘 못 하면 배포도 못 하고 원정대가 끝날 수 있다.
3. **API 명세 공동 설계에서의 역할**: 백엔드 구조를 모르는데 API 설계에 얼마나 의미 있게 참여할 수 있을지. "네가 말하는 대로 할게"가 되어버리면 학습 효과가 떨어질 것 같다.

### 가장 기대되는 것 (Top 3)

1. **실제 배포 경험**: 지금까지 로컬에서만 개발했는데, 다른 사람이 접속해서 쓸 수 있는 서비스를 만든다는 게 설렌다.
2. **백엔드 코드 읽기 경험**: 교차 코드 리뷰를 통해 백엔드 코드를 읽어볼 기회가 생긴다. 레벨3에서 백엔드 크루와 협업할 때 "그쪽 코드가 어떻게 생겼는지 전혀 모르겠다"는 상황을 미리 예방할 수 있을 것 같다.
3. **사용자 피드백을 받는 경험**: 직접 만든 서비스를 다른 크루가 써보고 의견을 준다는 것 자체가 새로운 경험이다. 그 피드백을 바탕으로 개선하는 유지보수 사이클도 실무에서 하는 것과 비슷해 보인다.
