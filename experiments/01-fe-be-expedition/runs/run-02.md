## 미션 설계서

## 1. 미션 개요

**한 줄 요약**: 프론트엔드 크루 1명과 백엔드 크루 1명이 페어로, 우테코 크루들이 실제 사용할 수 있는 웹 애플리케이션을 설계하고 개발하고 배포하고 유지보수한다.

### 핵심 경험

- **FE-BE 경계를 넘나드는 협업**: API 명세를 함께 설계하고, 서로의 코드를 리뷰하며, 웹 애플리케이션의 전체 흐름(요청 → 서버 → DB → 응답 → 렌더링)을 체감한다.
- **상대 영역의 테스트를 함께 작성**: 프론트엔드 크루가 백엔드 테스트를, 백엔드 크루가 프론트엔드 테스트를 **페어 프로그래밍으로 함께** 작성하며, 서로의 영역에서 테스트가 어떻게 작동하는지 경험한다.
- **배포 후 유지보수**: 실사용자의 피드백을 바탕으로 우선순위를 정하고, 개선 사이클을 돌린다.
- **레벨1 송곳의 확장**: 레벨1 원정대에서 집중 탐구한 기술적 주제(송곳)를 이번 미션에 의도적으로 적용하고, 페어에게 전파한다.

> **송곳이란?** 레벨1 원정대에서 각 크루가 스스로 정한 "집중 탐구 주제"를 뜻한다. 한 분야를 깊이 파고들어 자신만의 강점으로 만든 것이다. 예를 들어 레벨1에서 TDD를 송곳으로 삼은 크루가 있다면, 이번 미션에서 BE 크루에게 TDD 접근법을 전수하거나, 프로젝트 전체에 TDD 문화를 적용하는 식으로 송곳을 확장한다.

## 2. 단계별 구성

### 1단계: 기획 및 설계 (원정대 1~3일차)

**목적**: 무엇을 만들지 합의하고, FE-BE 간 계약(API 명세)을 확정한다.

**활동**:
- 페어가 함께 서비스 아이디어를 선정한다. 대상 사용자는 "우테코 크루"로 한정한다.
- 핵심 기능 3개 이내로 범위를 좁힌다.
- **API 명세를 아래 순서로 설계한다**:
  1. FE 크루가 화면별로 필요한 데이터 형태를 먼저 정의한다 (예: "이 화면에서는 `{id, title, author, createdAt}` 형태의 목록이 필요하다").
  2. BE 크루가 그 데이터를 제공할 수 있는 엔드포인트를 설계한다 (예: `GET /api/posts → [{id, title, author, createdAt}]`).
  3. 양쪽이 함께 요청/응답 형식을 조율하여 API 명세 문서를 확정한다.
- 데이터 모델을 함께 설계한다. FE 크루도 "이 데이터가 화면에서 어떻게 쓰이는지" 관점에서 ERD 작성에 참여한다.
- 레벨1 원정대에서 각자 만든 송곳을 공유하고, 이번 미션에 어떻게 적용할지 구체적 계획을 세운다.
- 기술 스택을 선정하고, **"왜 이 기술을 선택했는지"** 근거를 기획서에 기록한다.

**산출물**:
- 서비스 기획서 (1페이지: 대상 사용자, 핵심 기능, 해결하려는 문제, 기술 스택 선정 근거)
- API 명세 문서
- 데이터 모델 (ERD 또는 스키마)
- 송곳 적용 계획 (각자 1개: 무엇을 어떤 장면에서 적용할지)

**학습 체크포인트** (1단계 종료 시 자가 점검):
- [ ] FE-BE가 함께 API 명세를 설계하는 과정에서 상대 영역의 제약을 1개 이상 발견했는가?
- [ ] 레벨1 송곳을 이번 미션에 적용할 구체적 계획이 있는가?

### 2단계: 핵심 연결 (원정대 4~8일차)

**목적**: FE-BE가 실제로 연결되어 동작하는 것을 확인한다. **이 단계의 목표는 MVP 완성이 아니라, 핵심 1개 API가 FE에서 BE까지 관통하여 동작하는 것이다.**

> 예: "게시글 목록 조회" API 하나가 FE 화면에서 호출되어 BE 서버를 거쳐 DB 데이터를 가져와 화면에 표시되는 흐름을 완성한다.

**활동**:
- 핵심 기능 중 1개를 선택하여 FE-BE 연결을 우선 구현한다.
- 백엔드: 선택한 기능의 API를 구현한다. 테스트를 먼저 작성한 뒤 프로덕션 코드를 작성한다.
- 프론트엔드: 해당 기능의 UI를 구현한다. 1단계에서 합의한 API 명세를 기반으로 MSW 등을 활용해 독립적으로 개발을 시작하고, BE API가 준비되면 실제 연결한다.
- **교차 테스트 작성**: 페어 프로그래밍 방식으로 상대 영역의 테스트를 **함께** 작성한다. 해당 영역의 크루가 드라이버, 상대 영역 크루가 네비게이터가 되어 진행한다. 작성 후 "이 과정에서 새로 알게 된 것"을 각자 메모한다.
- 나머지 기능은 FE/BE 각각 독립적으로 개발을 진행한다 (3단계에서 통합).
- 매일 15분 이내의 페어 체크인을 진행한다: 진행 상황 공유 + API 변경 사항 동기화.

**산출물**:
- 핵심 1개 기능이 FE-BE 연결되어 동작하는 상태
- 교차 테스트 코드 + 교차 테스트 회고 메모 (각자 "새로 알게 된 것" 기록)
- 일일 체크인 기록 (간단한 메모)

**학습 체크포인트** (2단계 종료 시 자가 점검 + **코치 체크인**):
- [ ] 웹 요청이 FE → BE → DB → BE → FE로 흐르는 전체 과정을 설명할 수 있는가?
- [ ] 상대 영역의 테스트를 함께 작성하면서 배운 것이 1개 이상 있는가?

> **코치 체크인**: 2단계 종료 시점에 코치가 페어와 15분 면담을 진행한다. 위 체크포인트를 함께 확인하고, 3단계 진입 전 범위 조정이 필요한지 논의한다.

### 3단계: 기능 완성 및 배포 (원정대 9~12일차)

**목적**: 나머지 기능을 통합하고, 실제 환경에 배포하여 사용자 피드백을 수집한다.

> **원칙: 기능 완성보다 배포를 우선한다.** 모든 기능이 완성되지 않았더라도 핵심 기능이 동작하면 먼저 배포한다. 사용자 피드백을 수집할 수 있는 상태를 만드는 것이 이 단계의 최우선 목표다.

**활동**:
- 2단계에서 독립적으로 개발한 나머지 기능의 FE-BE 연결을 완료한다.
- FE/BE를 각각 배포한다. 배포 환경은 크루가 자유롭게 선택한다.
- 우테코 크루 5명 이상에게 서비스를 사용해달라고 요청하고, 구두 또는 서면으로 피드백을 수집한다.
- 수집한 피드백을 기능 개선, 버그 수정, UX 개선으로 분류한다.
- 원정대 발표에서 서비스 데모와 함께 기술적 의사결정 과정을 공유한다.

**산출물**:
- 배포된 서비스 URL (FE + BE)
- 사용자 피드백 정리 문서
- 원정대 발표 자료

**학습 체크포인트** (3단계 종료 시 자가 점검):
- [ ] BE 크루에게 FE 관련 기술적 질문을 받았을 때, 근거를 들어 설명한 경험이 있는가? (반대도 동일)
- [ ] 레벨1 송곳을 실제로 적용한 장면이 있는가?

### 4단계: 유지보수 (레벨2 남은 기간, 약 4~6주)

**목적**: 실사용자 피드백을 기반으로 서비스를 개선하며, 유지보수 경험을 쌓는다.

**활동**:
- 3단계에서 수집한 피드백과 자체 회고를 바탕으로 이슈 목록을 생성한다.
- 2주 단위 스프린트로 개선 작업을 진행한다. 매 스프린트 시작 시 이슈 우선순위를 함께 결정한다.
- 스프린트마다 최소 1회 배포한다.
- **스프린트 종료 시** 사용자 3명 이상에게 변경된 기능을 확인 요청하고, 새로운 피드백을 수집한다 (피드백 재수집).
- **중간 마일스톤**: 유지보수 2주차까지 "서비스를 1회 이상 사용한 크루 10명 달성"을 목표로 한다. 달성 여부에 관계없이 이 시점에 사용자 확보 전략을 회고한다.
- 마지막 스프린트 종료 시 회고 문서를 작성한다.

**산출물**:
- 이슈 목록 및 우선순위 (GitHub Issues 또는 유사 도구 활용)
- 스프린트별 변경 로그
- 최종 회고 문서

**학습 체크포인트** (최종):
- [ ] "피드백 수집 → 우선순위 결정 → 구현 → 배포 → 피드백 재수집" 사이클을 2회 이상 경험했는가?
- [ ] 이 미션을 통해 상대 영역(FE 또는 BE)과 기술적 대화를 할 수 있게 되었는가?

## 3. 요구사항

### 기능 요구사항

#### 필수

- 우테코 크루가 실제로 사용할 수 있는 웹 서비스를 개발한다.
- 최소 1개의 CRUD(Create, Read, Update, Delete) 기능을 포함한다.
- 프론트엔드와 백엔드가 HTTP API로 통신한다.
- 사용자가 데이터를 입력하고 저장된 결과를 확인할 수 있다.

#### 선택

- 사용자 인증/인가 기능
- 실시간 업데이트 (WebSocket 등)
- 외부 API 연동

### 비기능 요구사항

#### 필수

- **테스트**: 백엔드 API에 대한 테스트와 프론트엔드 주요 로직에 대한 테스트를 작성한다.
- **교차 테스트**: 상대 영역의 테스트를 **페어 프로그래밍으로 함께** 작성한다. 해당 영역 크루가 가이드하고, 상대 영역 크루가 직접 코드를 작성한다. 작성 후 각자 "이 과정에서 새로 알게 된 것"을 기록한다.
- **API 문서화**: API 명세를 README 또는 별도 문서에 기록한다.
- **배포**: 원정대 기간 내에 외부에서 접근 가능한 URL로 배포한다.
- **코드 리뷰**: 모든 PR에 대해 페어가 리뷰한다. FE 코드는 BE 크루가, BE 코드는 FE 크루가 리뷰한다.

#### 선택

- CI/CD 파이프라인 구축
- 에러 모니터링
- 성능 최적화

## 4. 협업 규칙

### 커뮤니케이션 구조

| 활동 | 빈도 | 형식 | 내용 |
|------|------|------|------|
| 페어 체크인 | 매일 1회 (15분 이내) | 대면 또는 화상 | 진행 상황, API 변경 사항, 막힌 점 공유 |
| 코드 리뷰 | PR마다 | GitHub PR 코멘트 | 상대 영역 코드 리뷰 (질문 위주도 가능) |
| 스프린트 계획 | 2주마다 | 대면 | 이슈 우선순위 결정, 작업 분배 |
| 중간 회고 | 원정대 종료 시 | 문서 작성 | 잘된 점, 개선할 점, 배운 점 |

### FE-BE 협업 포인트

1. **API 명세: FE가 먼저, BE가 이어서, 함께 확정**
   - FE 크루가 화면별로 필요한 데이터 형태를 먼저 정의한다.
   - BE 크루가 그 데이터를 제공할 수 있는 엔드포인트를 설계한다.
   - 양쪽이 함께 조율하여 최종 명세를 확정한다.
   - 이 순서를 지킴으로써 "BE가 주도하고 FE가 따라가는" 상황을 방지한다.

2. **교차 테스트: 함께 작성하며 배우기**
   - 상대 영역의 테스트를 페어 프로그래밍으로 함께 작성한다.
   - 해당 영역 크루가 드라이버(키보드를 잡는 사람), 상대 영역 크루가 네비게이터(방향을 안내하는 사람) 역할을 맡는다.
   - 완료 후 각자 "새로 알게 된 것"을 메모한다. 이 메모가 교차 테스트의 핵심 산출물이다.

3. **교차 코드 리뷰**: 상대 영역의 PR을 리뷰한다. 코드를 완벽히 이해하지 못해도, "이 부분이 무슨 역할인지 모르겠다"는 질문 자체가 가치 있다.

4. **장애 대응 공동 참여**: 배포 후 버그가 발생하면 FE/BE 중 어디 문제인지 함께 추적한다. 로그, 네트워크 탭, 에러 메시지를 함께 확인한다.

### 갈등 해결

- 기술적 의사결정에서 합의가 안 되면, 각자의 근거를 문서로 작성한 뒤 코치에게 의견을 구한다.
- 작업 속도 차이가 발생하면, 빠른 쪽이 느린 쪽의 작업을 페어 프로그래밍으로 돕는다.

## 6. 유지보수 가이드

### 유지보수 사이클

```
피드백 수집 → 이슈 등록 → 우선순위 결정 → 구현 → 코드 리뷰 → 배포 → 피드백 재수집
```

### 스프린트 운영

- 2주 단위 스프린트로 운영한다.
- 스프린트 시작 시 페어가 함께 이슈 우선순위를 정한다.
- 우선순위 기준: 사용자 영향도 > 기술 부채 > 새 기능.
- 스프린트마다 최소 1회 배포한다.
- **스프린트 종료 시** 사용자 3명 이상에게 변경된 기능을 확인 요청하고, 새로운 피드백을 수집한다.

### 피드백 채널

- 서비스 내 피드백 폼 (선택)
- 슬랙 채널에 사용 후기 요청 (필수)
- 주 1회 사용자 5명에게 직접 의견을 묻는다 (필수)

### 유지보수 기간의 학습 포인트

- **이슈 분류**: 사용자가 보고한 문제가 FE 이슈인지 BE 이슈인지 함께 판단한다.
- **회귀 방지**: 버그 수정 시 해당 케이스에 대한 테스트를 추가한다.
- **기술 부채 관리**: 원정대 기간에 급히 만든 코드를 리팩터링한다.

### 유지보수 종료 조건

- 레벨2 종료 시점에 최종 회고 문서를 작성한다.
- 회고에는 다음 항목을 포함한다: 전체 변경 이력 요약, 사용자 반응 추이, 기술적으로 배운 점, 협업에서 배운 점.

TDD/객체지향 → 2단계 "교차 테스트에서 배운 것". 송곳 연결 → 1단계 "적용 계획", 3단계 "실제 적용 장면". 각 기대효과가 우연이 아닌 구조에 의해 달성되도록 설계되어 있다. | 학습 체크포인트가 "자가 점검"에 머무른다. 코치가 중간에 확인하는 시점(예: 2단계 종료 시 코치 체크인)이 있으면 체크포인트의 실효성이 높아진다. |

| 유지보수 경험의 실질성 | 5 | run-01에서 피드백 루프가 1회성이었던 문제가 해결되었다. "스프린트 종료 시 사용자 3명 이상에게 변경된 기능 확인 요청"이 추가되어, "피드백 수집 → 구현 → 배포 → 피드백 재수집"의 반복 사이클이 구조적으로 보장된다. 학습 체크포인트에서도 "피드백 사이클을 2회 이상 경험했는가"를 명시적으로 점검한다. | 유지보수 기간이 4~6주로 길어 동기가 떨어질 수 있다. 중간 마일스톤(예: 2주차에 "사용자 10명 달성" 같은 구체적 목표)이 있으면 동기 유지에 도움이 된다. |

### 종합

- **총점: 25/25**
- **판정: 실험에 바로 사용 가능**
- **우선 개선 영역** (만점이지만, 실 운영 시 고려할 사항):
  1. **코치 체크인 시점 명시**: 학습 체크포인트가 자가 점검에 머무르므로, 2단계 종료 시점에 코치가 진행 상황을 확인하는 시점을 추가하면 체크포인트의 실효성이 높아진다.
  2. **3단계 분량 관리**: 기능 통합 + 배포 + 피드백 수집 + 발표 준비를 4일에 처리하는 것이 빠듯할 수 있으므로, "배포를 기능 완성보다 우선한다"는 가이드를 추가하면 더 안전하다.
  3. **유지보수 기간 동기 유지**: 4~6주의 유지보수 기간에 동기가 떨어질 수 있으므로, 중간 마일스톤(사용자 수 목표 등)을 설정하면 도움이 된다.

## Crew Perspective 피드백

### 첫 인상

백엔드 크루와 실제 서비스를 만든다는 건 여전히 설렌다. run-01보다 좋아진 건, "5일 만에 MVP를 만들라"는 압박이 사라지고 "먼저 1개 API만 연결해보자"로 바뀐 점이다. 해야 할 것이 여전히 많지만, 단계별로 뭘 해야 하는지가 더 구체적이라 막막함이 줄었다.

### 체크리스트 응답

| 항목 | 반응 | 구체적 지점 |
|------|------|------------|
| 이해도 | 명확 | 단계별 구성이 4단계로 나뉘어 있고, 각 단계의 목적과 산출물이 명시되어 있다. 특히 2단계의 "핵심 1개 API가 FE에서 BE까지 관통하여 동작하는 것"이라는 목표가 구체적이라 "어디까지 하면 되는 건지" 감이 온다. 송곳 설명도 추가되어서 처음 보는 크루도 이해할 수 있다. API 설계 순서(FE가 먼저 → BE가 이어서 → 함께 확정)도 명확하다. |
| 동기 | 재밌겠다 | 실제 크루들이 쓸 서비스를 배포한다는 점이 여전히 가장 큰 동기다. "기술 스택 선정 근거를 기록하라"는 요구사항도 괜찮다. 왜 React를 선택했는지 스스로 정리해보는 건 의미가 있다. |
| 불안 | 낮음~중간 | 교차 테스트가 "1개 이상 작성"에서 "페어 프로그래밍으로 함께 작성"으로 바뀐 게 오히려 안심된다. 혼자서 JUnit을 써야 하는 게 아니라 BE 크루가 옆에서 가이드해준다는 구조이니까. 다만 3단계(4일)에서 나머지 기능 통합 + 배포 + 피드백 수집 + 발표까지 해야 하는 건 여전히 빠듯해 보인다. |
| 협업 걱정 | 낮음 | API 명세 설계 순서가 "FE가 먼저 데이터 형태를 정의한다"로 명시된 게 가장 큰 변화다. run-01에서는 "공동 설계"라고만 해서 결국 BE가 주도하지 않을까 걱정했는데, 이제는 내가 먼저 "이 화면에는 이런 데이터가 필요하다"를 말하고 시작하는 구조라 역할이 분명하다. |

### 가장 걱정되는 것 (Top 3)

1. **3단계 일정**: 나머지 기능 통합 + 배포 + 사용자 피드백 수집 + 발표 자료 준비를 4일에 끝내야 한다. 2단계에서 독립적으로 개발한 코드를 통합하면서 예상치 못한 문제가 생길 수 있다. 기능을 줄이는 게 나을 수도 있다.
2. **유지보수 기간의 동기 유지**: 원정대 기간에는 집중해서 하겠지만, 레벨2 남은 기간(4~6주) 동안 다른 미션도 있을 텐데 이 프로젝트에 계속 시간을 쓸 수 있을지 걱정된다. 스프린트마다 배포 + 피드백 재수집을 해야 하는 것도 부담이다.
3. **학습 체크포인트의 자가 점검**: 체크포인트가 있는 건 좋은데, 스스로 "예/아니오"를 체크하는 것만으로 충분한지 모르겠다. 코치가 확인해주는 시점이 있으면 더 안심될 것 같다.

### 가장 기대되는 것 (Top 3)

1. **1개 API 관통 경험**: 2단계에서 "게시글 목록 조회" 같은 1개 기능으로 FE → BE → DB 전체 흐름을 먼저 경험하는 구조가 좋다. 이걸 해보면 나머지 기능도 같은 패턴으로 확장할 수 있을 것 같다.
2. **API 설계에서 FE의 역할**: "화면에 필요한 데이터 형태를 내가 먼저 정의한다"는 게 기대된다. 지금까지는 BE가 주는 데이터에 맞춰 UI를 만들었는데, 반대로 "UI에 이런 데이터가 필요하다"를 먼저 말하는 경험이 새롭다.
3. **교차 테스트 페어 프로그래밍**: BE 크루가 옆에서 가이드하면서 함께 테스트를 작성하는 구조가 안심되고 기대된다. 혼자 BE 테스트를 쓰라고 하면 막막했을 텐데, 함께 하면서 배울 수 있다는 게 좋다.
