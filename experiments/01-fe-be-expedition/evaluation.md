# 실험 01 평가: FE-BE 원정대 미션 설계

## 1. 실험 결과 요약

| 항목 | 결과 |
|------|------|
| 총 회차 | 2회 (run-01 → run-02) |
| 최종 루브릭 점수 | **22/25** (사람 채점) |
| 에이전트 자체 채점 | 25/25 (과대 평가, 3점 차이) |
| 성공 판정 | **성공** — 3회 이내, 20점 이상, 전 차원 3점 이상 |
| 운영 가능성 | 최종 보완 3건 반영 후 **크루에게 전달 가능한 수준** |

## 2. 회차별 점수 추이

| 차원 | run-01 | run-02 | 변화 |
|------|--------|--------|------|
| 협업 구조의 현실성 | 4 | **5** | +1 (FE→BE→조율 순서 명시) |
| 학습 목표 달성 가능성 | 4 | **4** | 0 (체크포인트 추가했으나 자가 점검 한계) |
| 단계 설계의 적절성 | 4 | **4** | 0 (2단계 개선, 3단계 여전히 빠듯) |
| 요구사항 구체성/자율성 균형 | 4 | **5** | +1 (기술 스택 선정 근거 기록 추가) |
| 유지보수 경험의 실질성 | 4 | **4** | 0 (피드백 재수집 추가, 동기 유지 장치 부족) |
| **총점** | **20** | **22** | **+2** |

개선이 유효했다: 5개 개선 포인트 중 2개 차원에서 실제 점수 상승.

## 3. 운영 가능성 판단

| 질문 | 답변 |
|------|------|
| 크루에게 그대로 전달해도 "뭘 해야 하는지" 이해할 수 있는가? | **Yes** — 단계별 목적/산출물이 명확하고, 송곳 설명, API 설계 순서 등 모호했던 부분이 보완됨 |
| 코치가 추가 보충 설명 없이 미션을 시작할 수 있는가? | **거의 Yes** — 코치 체크인 시점, 3단계 배포 우선 원칙, 유지보수 중간 마일스톤을 최종 반영 완료 |
| 기간 내 완료 가능한 분량인가? | **Yes** — 2단계 목표를 "1개 API 관통"으로 현실화. 3단계는 "배포 우선" 가이드 추가로 범위 관리 가능 |

## 4. 에이전트별 기여도

| 에이전트 | 유용했던 부분 | 한계 |
|---------|-------------|------|
| **Mission Designer** | 7개 섹션을 빠짐없이 구조화, 일관된 문체, 구체적 산출물 목록 | 일정 현실성 판단 약함 (5일 MVP), 교차 테스트 수량 기준으로 형식화 |
| **Coach Reviewer** | 루브릭 기반 체계적 평가, 각 차원별 개선 포인트가 구체적 | 자기 생성물에 대한 평가에서 관대 (run-02에서 만점) |
| **Crew Perspective** | **가장 유용.** "송곳이 뭔지 모르겠다", "API 설계에서 내 역할이 뭐지", "코치 확인이 있으면 좋겠다" 등 코치가 놓치기 쉬운 수강생 체감 문제를 정확히 짚음 | 불안 수준 변화가 다소 낙관적 |
| **반복개선 (design.md)** | 실험 설계서의 구조적 완성도를 크게 높임 (가설 근거, 실패 분석 프레임, run 템플릿) | 실험 설계서 개선과 실험 실행이 병렬이라 run-01이 개선된 템플릿을 반영하지 못함 |

## 5. 인사이트

### 에이전트 활용에 대해

1. **에이전트가 자기 산출물을 평가하면 과대 평가한다.** run-02에서 에이전트 25점 vs 사람 22점. 평가 에이전트는 생성 에이전트와 분리하거나, 사람이 최종 채점해야 한다.

2. **Crew Perspective(수강생 관점)가 가장 높은 ROI.** 코치는 교육 설계자 관점에 갇히기 쉬운데, "이거 읽으면 뭘 해야 하는지 모르겠다"는 피드백이 가장 직접적인 품질 지표다.

3. **1회 만에 "사용 가능" 등급(20점) 도달.** 에이전트의 구조화 능력은 충분하지만, "현실성 판단"은 사람의 개입이 필수적이다. 최적 워크플로우: 에이전트가 구조 생성 → 사람이 현실성 검증 → 에이전트가 반영.

### 루브릭에 대해

4. **"학습 목표 달성 가능성"과 "유지보수 실질성"이 4점에 머문 이유**: 둘 다 "시간이 지나야 확인되는 차원"이다. 설계서만으로는 5점을 주기 어렵고, 실제 운영 후에야 판단 가능하다. 루브릭에 이 한계를 명시할 필요가 있다.

### 프로세스에 대해

5. **design.md 개선과 run 실행을 병렬로 돌리면 동기화 문제가 생긴다.** run-01은 개선 전 design.md 기준으로 만들어져 run 파일 템플릿을 따르지 않았다. 다음 실험에서는 design.md를 먼저 확정한 후 run을 시작해야 한다.
