# FE-BE 원정대 미션 설계 평가 루브릭

## 배경

프론트엔드 1명 + 백엔드 1명이 페어로 웹앱을 만들어 배포하고, 이후 유지보수/개선하는 미션.

## 평가 차원 (5개)

### 1. 협업 구조의 현실성 (Collaboration Feasibility)

FE-BE 크루가 실제로 함께 일할 수 있는 구조인가?

| 점수 | 기준 |
|------|------|
| 1 | FE/BE 역할 구분이 없거나, 한쪽이 놀게 되는 구조 |
| 2 | 역할은 나뉘지만 접점이 거의 없어 각자 개발하는 구조 |
| 3 | API 계약 등 명시적 접점이 있으나 협업 빈도가 낮음 |
| 4 | 개발 전 과정에서 자연스러운 협업 포인트가 설계됨 |
| 5 | 협업 없이는 진행 불가능한 구조 + 서로의 영역을 이해해야 하는 장치 포함 |

### 2. 학습 목표 달성 가능성 (Learning Alignment)

기대효과에 명시된 학습 목표를 실제로 달성할 수 있는가?

| 점수 | 기준 |
|------|------|
| 1 | 기대효과와 미션 내용이 거의 무관함 |
| 2 | 일부 기대효과만 간접적으로 달성 가능 |
| 3 | 대부분의 기대효과를 달성할 수 있으나 우연에 의존 |
| 4 | 기대효과 달성을 유도하는 구조가 미션에 내장됨 |
| 5 | 각 기대효과에 대응하는 명시적 활동/체크포인트가 설계됨 |

기대효과 체크리스트:
- [ ] 웹 애플리케이션 전체를 이해하는 개발자 성장
- [ ] 백엔드 구조를 이해하고 대화할 수 있는 역량
- [ ] TDD 습관 → 레벨3 이후 코드 품질 기반
- [ ] 레벨1 원정대 "송곳"과의 연결고리
- [ ] FE만으로 경험하기 어려운 TDD/객체지향 고민

### 3. 단계 설계의 적절성 (Phase Design)

초기 배포 → 유지보수/개선의 흐름이 자연스럽고 학습 곡선에 맞는가?

| 점수 | 기준 |
|------|------|
| 1 | 단계 구분 없이 "만들어라"만 있음 |
| 2 | 단계는 있으나 각 단계의 목적/산출물이 불명확 |
| 3 | 단계별 목적은 있으나 시간 배분이 비현실적 |
| 4 | 단계별 목적/산출물/시간이 적절하고 점진적 난이도 상승 |
| 5 | 각 단계에서 이전 단계의 학습이 자연스럽게 활용되는 설계 |

### 4. 요구사항의 구체성과 자율성 균형 (Specification Balance)

너무 구체적이면 자율성이 없고, 너무 추상적이면 방향을 잃는다.

| 점수 | 기준 |
|------|------|
| 1 | "웹앱을 만들어라" 수준의 과도한 추상화 |
| 2 | 기술 스택/기능은 지정되나 "왜"가 없음 |
| 3 | 제약 조건은 있으나 크루의 선택지가 부족 |
| 4 | 핵심 제약은 명확 + 크루가 결정할 영역이 설계됨 |
| 5 | 제약/자율의 경계가 명확하고, 자율 영역에서의 의사결정이 학습으로 이어지는 구조 |

### 5. 유지보수 경험의 실질성 (Maintenance Authenticity)

배포 이후 유지보수/개선이 "진짜 경험"이 되는가?

| 점수 | 기준 |
|------|------|
| 1 | 배포 후 유지보수 활동이 정의되지 않음 |
| 2 | "개선하라"는 지시만 있고 동기/피드백 루프 없음 |
| 3 | 유지보수 과제가 있으나 인위적 (실사용자 없음) |
| 4 | 실사용자 피드백 or 실제 이슈 기반의 유지보수 구조 |
| 5 | 사용자 피드백 → 우선순위 결정 → 구현 → 배포의 사이클이 설계됨 |

## 종합 판정

| 총점 | 판정 |
|------|------|
| 20-25 | 실험에 바로 사용 가능 |
| 15-19 | 특정 차원 보완 후 사용 가능 |
| 10-14 | 구조적 재설계 필요 |
| 5-9 | 방향 재검토 필요 |
